## Digital Twin Bidirecional - Conceito e Arquitetura

### Conceito Identificado: Digital Twin Prototype (DTP)

**Definição:** Digital Twin Prototype é o conceito de criar o gêmeo digital **antes** do hardware físico existir. Isso valida exatamente a ideia do Cássio: criar dois ambientes simulados onde um funciona como "mundo físico" e o outro como Digital Twin.

**Arquitetura Proposta:**
```
┌─────────────────────────────────────────────────────────────────────┐
│                  DIGITAL TWIN BIDIRECIONAL                          │
│                                                                      │
│  ┌─────────────────┐         MAVLink/UDP         ┌───────────────┐ │
│  │   AMBIENTE A    │ ◄─────────────────────────► │  AMBIENTE B   │ │
│  │ (Mundo "Físico")│        Sincronização        │ (Digital Twin)│ │
│  │                 │         ~20ms latência       │               │ │
│  │  Gazebo #1      │                              │  Gazebo #2    │ │
│  │  + ArduCopter   │                              │  + ArduCopter │ │
│  │  SITL #1        │                              │  SITL #2      │ │
│  │  SYSID: 1       │                              │  SYSID: 2     │ │
│  └─────────────────┘                              └───────────────┘ │
│                                                                      │
│  Quando mexer em QUALQUER drone em QUALQUER ambiente,              │
│  o comportamento é refletido no ambiente oposto.                    │
└─────────────────────────────────────────────────────────────────────┘
```

### Componentes Necessários

1. **Gazebo Harmonic** - Simulador físico (2 instâncias ou 1 instância com 2 mundos)
2. **ArduPilot SITL** - Controladores de voo em software (2 instâncias)
3. **MAVLink Bridge** - Protocolo de comunicação entre simuladores
4. **Sincronização de Estados** - Bridge Python para propagar estados

### Requisitos Técnicos Identificados

**Requisitos Funcionais:**
- RF01: Sincronização bidirecional de posição, orientação, velocidade
- RF02: Latência máxima de sincronização: 20-50ms (aceitável para DT)
- RF03: Propagação de comandos de controle entre ambientes
- RF04: Detecção de anomalias e divergências entre estados

**Requisitos Não-Funcionais:**
- RNF01: Latência de sincronização < 50ms (referência: 20ms em paper de DT)
- RNF02: Taxa de atualização: 50Hz mínimo (típico de controle de voo)
- RNF03: Suportar até 5 drones simultâneos por ambiente

**Requisitos de Domínio:**
- RD01: Modelo de drone compatível (Crazyflie, quadrotor genérico)
- RD02: Física realista para demonstração de DT
- RD03: Interface visual para observação lado-a-lado

- Protocolo MAVLink sobre UDP é adequado

## TraceTheory - Arqueologia Intelectual Interativa (17:00-19:55 GMT-3)

### Projeto: Ferramenta de Arqueologia Intelectual

**Objetivo:** Traceback recursivo da origem da computação a partir do paper de Turing (1936), coletando referências citadas e citações recebidas.

### Componentes Desenvolvidos

#### 1. GROBID - Extração de Referências de PDFs

**Instalação:**
- Container Docker: `docker.io/grobid/grobid:0.8.0` via Podman
- Porta: 8070
- API verificada como ativa: `http://localhost:8070/api/isalive → true`

**Endpoints Utilizados:**
- `POST /api/processHeaderDocument` - Extrai metadados do cabeçalho
- `POST /api/processReferences` - Extrai seção de referências
- `POST /api/processFulltextDocument` - Extrai texto completo estruturado

**Resultado para Turing 1936:**
- Header: 26 bytes (vazio - paper antigo sem metadados formais)
- References: 0 bytes (vazio - paper antigo sem seção formal de referências)
- Fulltext: 75,399 bytes (texto completo estruturado em TEI/XML)

**Limitação:** GROBID não funciona bem em papers de 1936 porque:
1. Não têm seção formal de referências
2. Referências estão incorporadas no texto (footnotes)
3. Formato de citação não-padronizado

#### 2. Vintage Reference Extractor

**Arquivo:** `scripts/vintage_reference_extractor.py`

**Abordagem:** Regex customizado para papers antigos (1900-1950):
- Detecção de footnotes explícitos (f, X, *)
- Detecção de menções inline de autores (Church, Hilbert, Gödel, Kleene)
- Parsers específicos para formatos da época

**Resultado para Turing 1936:**
- 8 referências extraídas manualmente
- Fontes: footnotes e menções no texto
- Inclui: Gödel (1931), Church (1936a/b), Hilbert & Ackermann (1928), etc.

#### 3. Grafo de Citações com Metadados

**Arquivos gerados:**
- `output/visualization_graph.json` - Grafo completo para D3.js
- `output/citation_graph_full.json` - Grafo processado pelo motor
- `output/turing_1936_backward_citations.json` - Referências detalhadas
- `output/turing_1936_backward_citations.bib` - Formato BibTeX

**Dados extraídos via OpenAlex API:**
- DOIs, OpenAlex IDs, URLs
- Contagem de citações
- Metadados completos (autores, venue, ano)

#### 4. Interface Web Interativa

**Arquivo:** `output/index.html`

**Funcionalidades:**
- Visualização D3.js force-directed graph
- Nós coloridos por tipo (original, correção, livro, backward/forward citation)
- Tooltips com metadados (título, autores, ano, DOI, PDF)
- Sidebar com detalhes ao clicar
- Exportação JSON
- Zoom e pan

#### 5. Motor de Recursão

**Arquivo:** `scripts/citation_graph.py`

**Classe CitationGraph:**
- `fetch_openalex_work(openalex_id)` - Busca metadados via API
- `fetch_backward_citations(openalex_id)` - Referências que o paper cita
